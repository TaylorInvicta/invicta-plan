The Financial Connection

Progress tracking is the missing denominator

How tracking progress answers the money question
1
Real % complete → accurate claims
You can only claim what you know is done. Right now claims are estimates — and estimates get disputed or underclaimed.
Concrete example
1% more accurately claimed on Logan = ~$21k more revenue per claim cycle
2
Progress % vs budget spend → margin visible now
The run chase graph exists today — it shows cost vs budget. What it can't show is whether that spend is proportionate to progress. Adding completion % as a line makes the gap visible while there's still time to act.
Concrete example
East Brisbane: 42% complete, 89% budget used. That's visible at 42% — not at 95% when it's too late.
3
Defensible claims → fewer disputes, faster payment
Builders dispute estimates. They can't dispute real data — a QA checklist, a weekly install log, a scope item at 78% with Al's daily updates behind it.
Concrete example
East Brisbane CA disputes 30% claim. With evidence: claim stands. Without: negotiation, delay, maybe reduction.
4
PM-managed recap → claims go out on time, every month
Taylor submits late because the manual prep is too heavy. PM-managed recap with auto-populated % makes it light enough to submit on the 31st. Consistently.
Concrete example
Claiming 2 weeks earlier each month at current project values = $100k+ earlier in the bank per month.
5
Dashboard closes the loop
All four of the above — progress %, margin %, claimed vs burn, project health — on one screen. Monday morning. No phone calls. Taylor reads it in 2 minutes.
The answer
"28.1% average margin. 6 of 8 on track. East Brisbane at risk." That's "are we making money?" answered.

Pre-PC items (louvers, glazing) skip Powder Coat — go directly Fabricating → Ready for Delivery. Set at scope item creation, editable any time.

Partial batchesv2 — a scope item often goes to Yates or to site across more than one load. The stage only advances when the last unit of that item has moved; until then the fab / install % carries the truth. A half-shipped item stays Fabricating with its % climbing — it never falsely shows Powder Coat.

On Holdv2 — when a drawing revision (Rev 01) lands after IFC, the PM flags the item On Hold in Connect. It shows immediately in Al's Factory View, replacing the phone-call-and-pull-the-pack-off-the-floor routine. Cleared when the revised pack is reissued.

Stages can move backwardsv2 — rework and defects happen. Any stage change, forwards or back, is logged with who and when.

Defect scope itemsv2 — defects get created as their own scope item type, drawing on a per-project defect allowance that covers drafting, fab, or install changes. Invicta sees maybe 20–30 defects a project (other trades see thousands) — so it's lightweight, but it's accounted for instead of silently eating the original item's hours.

Variation guardv2 — all contract scope items are created at signing, so any item added later is almost by definition a variation. Adding one mid-project prompts "Is this a variation?" — defaulting to yes. Closes the gap where variations get typed in as plain scope items in the rush of the day.

Existing projects migrate toov2 — Logan already matches the target structure. East Brisbane, Seaworld, and the other live jobs get restructured to it during week 2, by the PMs with support. New projects start right; live ones catch up.

Straight talk on week 1–2 datav2 — Phase 1 progress bars run on stage + drafting data. The real fab % lands in Phase 2 and install % in Phase 3. The structure comes first; the numbers fill in behind it.

Old Workflow board retiresv2 — the abandoned five-status kanban still showing on scope item pages comes down in Phase 1. Its lifecycle statuses are superseded by the pipeline; its task table is kept for the Asana-replacement build.

How the claim % is calculated
Added in v2

Fab % and install % are two numbers — the recap needs one. Each scope item's overall % is a value-weighted blend of its stages, so the claim is defensible line by line and early-stage work (drawings issued, materials ordered) still claims what current practice allows.

Worked example — BAL01 Gates & Frames (welded, supply + install)
StageWeightStatusEarns
Drafting (IFC issued)10%Done10%
Materials ordered15%PO sent15%
Fabrication35%60% (Al's daily log)21%
Delivered to site5%Not yet0%
Install35%Not started0%
Overall % — auto-filled into the recap46%

Weights are starting defaults per item type — Taylor tunes them once per type (welded / pre-PC / supply-only) and they apply everywhere. Supply-only items carry no install weight and finish at Delivered.

Project progress is value-weighted too — a $200k scope item moves the project bar further than a $6k one. No misleading simple averages.

PMs can override any auto % in the recap — the override is flagged to Taylor with the variance shown, so judgment stays possible but visible.

Rollout

10 weeks — one phase at a time

Each phase beds in before the next starts. The architecture is built holistically. The activation is layered.

Weeks 1–2

Phase 1 — Foundation

Scope structure + stage pipeline. PM setup only.

Weeks 3–4

Phase 2 — Fab Tracking

Al's daily % · QA sign-off · Powder Coat auto-trigger.

Weeks 5–6

Phase 3 — Install Tracking

Friday install log inside Quick Log.

Weeks 7–10

Phase 4 — Claims & Dashboard

PM recap · Taylor approves · Dashboard live.

Design principle: Holistic architecture, layered activation. The full system is designed as one connected piece — scope structure, fab inputs, install inputs, claims, and leadership view all built together. Activated in stages so the team adapts to one new habit at a time.

Phases gate on the habit, not the calendarv2 — Phase 3 starts once Al's daily % has held for two straight weeks; Phase 4 once the Friday install log has stuck the same way. Ten weeks is the target, not a forced march — a week's slip to let a habit bed in is cheaper than a dashboard nobody trusts.
